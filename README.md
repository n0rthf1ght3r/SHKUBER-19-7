## 2. Учет системных расходов (Overhead) и выбор типа нод

Рабочие ноды не могут отдать 100% ресурсов подам. Ресурсы требуются для:
1.  **OS Kernel & System Services** (SSH, Systemd и т.д.).
2.  **Kubelet** (агент узла).
3.  **Container Runtime** (Containerd/Docker).
4.  **System Pods** (CNI plugin, kube-proxy, CoreDNS, logs collector, monitoring agent).

**Стратегия резервирования:**
* Закладываем ~20-25% RAM и ~10-15% CPU на системные нужды.
* **Принцип N+1:** Кластер должен выдержать полную нагрузку (17 CPU / 30.25 GB), даже если одна нода полностью выйдет из строя.

### Выбор конфигурации ноды (Worker Node Flavor)
Нам нужно размещать "тяжелые" поды (БД и Кэш), требующие 4 ГБ RAM и 1 CPU.
* Маленькие ноды (2 CPU / 4 GB RAM) не подойдут: один под БД займет всю память, не оставив места системе, либо вообще не запустится.
* Оптимально, чтобы нода вмещала 2-3 крупных пода + системные расходы.

**Выбранная конфигурация:**
* **Flavor:** Standard-2x (условное название)
* **CPU:** 8 vCPU
* **RAM:** 16 ГБ

*Доступные ресурсы на такой ноде (примерно):*
* ~7 vCPU (после вычета системы)
* ~12-13 ГБ RAM (после вычета системы)

## 3. Расчет количества рабочих нод

### Расчет для сценария "Без сбоев"
Нам нужно разместить **17 vCPU** и **30.25 ГБ RAM**.
Попробуем распределить это на ноды конфигурации (8 CPU / 16 GB -> Allocatable: ~7 CPU / ~13 GB):

1.  **По CPU:** $17 / 7 \approx 2.4$ -> Нужно минимум **3 ноды**.
2.  **По RAM:** $30.25 / 13 \approx 2.3$ -> Нужно минимум **3 ноды**.

*Проверка:* На 3 нодах (суммарно ~39 GB RAM allocatable) нагрузка в 30.25 GB помещается комфортнo.

### Расчет с учетом отказа (N+1)
Если у нас будет всего 3 ноды и одна выйдет из строя, оставшиеся 2 ноды (суммарно ~14 CPU / ~26 GB RAM) **не вытянут** нагрузку в 17 CPU / 30.25 GB RAM. Приложение начнет тормозить или падать.

Поэтому добавляем **+1 ноду** для горячего резерва.

**Итого:** 4 ноды (8 CPU / 16 GB).

*Проверка отказа:*
* Всего ресурсов: 4 ноды.
* Минус 1 нода = 3 ноды.
* Емкость 3 нод: ~21 CPU / ~39 GB RAM.
* Нагрузка: 17 CPU / 30.25 GB RAM.
* **Результат:** Нагрузка помещается с запасом.

## 4. Требования к Control Plane (Master Nodes)

Так как проект требует отказоустойчивости для БД и Кэша, сам Kubernetes тоже должен быть отказоустойчивым.
Для кворума etcd требуется нечетное количество мастер-нод, минимум 3.

Нагрузка на Control Plane зависит от количества объектов. Для 21 пода нагрузка минимальна.

**Конфигурация Master Node:**
* Количество: **3 шт.**
* Ресурсы: **2 vCPU / 4 ГБ RAM** (минимум для стабильной работы etcd).

## 5. Итоговая спецификация кластера

Для обеспечения работы проекта с учетом резервирования N+1 и системных расходов необходимы следующие ресурсы:

### Worker Nodes
Предназначены для запуска приложений.

* **Количество:** 4 шт.
* **CPU:** 8 vCPU (на каждую)
* **RAM:** 16 ГБ (на каждую)
* **Disk:** SSD 50-100 ГБ (OS + Docker Images + Logs). Данные БД рекомендуется хранить на внешних томах.

### Control Plane
Предназначены для API Server, Scheduler, Controller Manager, Etcd.

* **Количество:** 3 шт.
* **CPU:** 2 vCPU (на каждую)
* **RAM:** 4 ГБ (на каждую)

### Суммарные ресурсы "железа" (Total Capacity)
* **Всего ядер:** $(4 \times 8) + (3 \times 2) = 38$ vCPU.
* **Всего памяти:** $(4 \times 16) + (3 \times 4) = 76$ ГБ RAM.
